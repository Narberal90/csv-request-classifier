# CSV Request Classifier

Автоматично класифікує внутрішні запити команд за допомогою Gemini LLM.
Завантажте CSV-файл вхідних повідомлень — отримайте структурований JSON та підсумковий звіт у форматі Markdown.
---

## Setup

### 1. Клонуйте репозиторій та встановіть залежності

```bash
git clone https://github.com/Narberal90/csv-request-classifier.git
cd CSV_request_analizer
```

### 2. Налаштуйте змінні середовища

```bash
cp .env.example .env
# Відредагуйте .env та вставте свій Gemini API key
```

Отримати безкоштовний ключ можна тут https://aistudio.google.com/apikey

---

## Запуск

```bash
docker build -t request-classifier .
docker run --env-file .env -p 8000:8000 request-classifier
```

Сервіс буде доступний за адресою `http://localhost:8000`. Інтерактивна документація `http://localhost:8000/docs`.

---

## Ендпоінти

### `GET /health`

```bash
curl http://localhost:8000/health
# {"status": "ok"}
```

### `POST /classify-file`

Завантажте CSV-файл із колонками: `id, channel, timestamp, raw_text`.

```bash
curl -X POST http://localhost:8000/classify-file \
  -F "file=@input_requests.csv"
```
або використайте інтерактивну документацію для POST запиту

Response:

```json
{
  "total_rows": 20,
  "classified": 19,
  "failed": 1,
  "results": [...],
  "errors": [...],
  "report": "# Request Classification Report\n..."
}
```

Також у директорію `src/` записуються `output.json` та `report.md`.

## Схема вихідних даних

Кожен класифікований запит у `results` має такий вигляд:

```json
{
  "id": "42",
  "channel": "slack",
  "timestamp": "2024-06-01 10:23",
  "category": "звіт/аналітика",
  "target_department": "продажі",
  "priority": "medium",
  "short_summary": "Потрібен щотижневий звіт по конверсії лідів у CRM.",
  "requested_actions": ["підключити CRM", "налаштувати автозвіт"],
  "needs_clarification": false,
  "confidence": "high"
}
```

### Навіщо було додано поле `confidence`

Рівень впевненості моделі у власній класифікації є корисним сигналом, особливо якщо сервіс інтегрований із frontend-додатком. Результати з низькою впевненістю можна візуально виділяти для ручної перевірки, щоб оператор одразу бачив сумнівні класифікації без необхідності переглядати кожен запис.

## Обмеження та потенційні проблеми рішення

### Некоректна відповідь LLM
Gemini отримує `response_json_schema`, що жорстко обмежує формат відповіді. Якщо модель все ж поверне некоректний JSON або порушить enum-обмеження, це буде перехоплено валідацією Pydantic — запит потрапить до `errors` і буде позначений у звіті. Без падіння сервісу та без прихованої втрати даних.

### Недетермінованість
LLM-моделі є недетермінованими — два запуски на однакових даних можуть дати різні результати. Для зменшення цього ефекту системний промпт містить явні правила класифікації з прив'язкою до ключових слів (`терміново` → `high` тощо).

### Ліміт
`gemini-2.5-flash` Безкоштовний тариф gemini-2.5-flash обмежений. Сервіс враховує це через semaphore (максимум 3 паралельні запити) та повторні спроби з інтервалами 10с → 30с → 60с для будь-яких тимчасових помилок.

### Великі файли
CSV-файл повністю завантажується в пам'ять. Для файлів розміром понад 100 MB доцільніше використовувати потокове читання.

## Як можна було б покращити:

1. **Резервні моделі (Model fallback)** — якщо основна модель недоступна після всіх повторних спроб, автоматично переходити до альтернативної (наприклад, gemini-2.5-flash-lite), зберігаючи назву використаної моделі в кожному результаті для відстеження можливих відмінностей у якості.


2. **Сповіщення для команди** — за наявності бази співробітників автоматично направляти класифікований запит відповідній людині в цільовій команді ще до формального призначення задачі. Це може допомогти цій людині краще організувати свій час.

3. **Frontend із візуалізацією** — відображати поле у зрозумілому вигляді: результати з високою впевненістю автоматично потрапляють у чергу обробки, а результати з низькою впевненістю позначаються для додаткової перевірки людиною.
